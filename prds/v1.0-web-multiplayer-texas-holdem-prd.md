# Web 多人在线德州扑克产品需求文档（PRD）

- 文档版本：v1.0
- 产品阶段：MVP / 可联机验证版本
- 更新时间：2026-03-14
- 适用对象：产品、设计、前端、后端、测试、部署维护

## 1. 产品定位

本产品是一款基于网页的多人在线德州扑克游戏，支持桌面端与手机端访问。玩家可以通过同一房间名进入同一牌桌，完成创建房间、加入房间、观战、入座、发牌、下注、弃牌、摊牌、结算等完整对局流程。

当前阶段的目标不是上线完整商业化扑克平台，而是交付一个具备真实多人联机能力、可持续迭代、规则相对完整、可部署到公网服务器的 Web 德州扑克 MVP。

## 2. 当前阶段程序输出

当前版本已经可以输出一个可运行的多人在线德州扑克程序，包含以下部分：

### 2.1 后端能力

- 基于 Go 实现房间管理、玩家连接管理与牌局引擎
- 支持 SSE 实时状态下发 + HTTP Action 上行交互
- 支持 `classic` 与 `short deck` 两种模式
- 支持多人牌局、盲注、发牌、下注轮、摊牌与结算
- 支持 side pot（边池）与多人 all-in 结算
- 支持 Host 权限控制
- 支持局间移除玩家、解散房间、空房间自动销毁

### 2.2 前端能力

- 基于 React + Vite + TypeScript 实现牌桌 UI
- 支持登录页、牌桌页、观众席、入座与换座
- 支持展示：
  - 房间信息
  - Host 信息
  - 底池
  - 盲注位置
  - 玩家筹码
  - 当前下注额
  - 公共牌 / 手牌
  - 当前轮次
  - 胜利结算信息
- 支持移动端与 PC 端响应式布局
- 支持当前行动玩家高亮提示
- 支持移动端底部固定操作区

### 2.3 部署能力

- 后端可通过 Docker 镜像部署到 Azure Container Apps
- 前端可通过 Vercel 或静态站部署
- 已提供前后端 Dockerfile，可用于本地和线上构建

## 3. 产品目标

### 3.1 业务目标

- 验证多人网页德州扑克的基础联机体验
- 建立可扩展的游戏后端架构
- 建立后续迭代的产品与技术基线

### 3.2 用户目标

- 玩家能够低门槛创建或加入房间
- 玩家能够在电脑和手机上完成一局完整德州扑克
- 玩家能够明显识别当前轮到谁行动
- 房主能够管理房间，避免异常玩家影响对局

## 4. 目标用户

- 熟悉德州扑克规则的朋友局玩家
- 需要快速开局的私人牌桌组织者
- 需要跨设备联机体验的轻量游戏用户
- 项目开发者、测试人员和演示使用者

## 5. 核心使用场景

### 场景 1：创建并开始一局经典德州

1. 玩家 A 进入网页，填写房间名、UID、昵称、买入码量
2. 玩家 A 创建房间并成为 Host
3. 玩家 B / C / D 使用相同房间名进入房间
4. Host 选择 `classic` 模式并点击 `Start Hand`
5. 玩家完成发牌、下注、亮牌、结算

### 场景 2：短牌模式游戏

1. Host 选择 `short deck`
2. 系统移除 `2/3/4/5`
3. 按短牌规则进行发牌、比牌与下注顺位处理
4. 短牌模式下注轮从 `SB` 开始

### 场景 3：中途加入与观战

1. 新玩家在牌局进行中加入房间
2. 系统先将其置于观众席
3. 当前牌局不被阻塞
4. 该玩家可选择下局加入牌桌

### 场景 4：房间治理

1. 某玩家掉线或异常残留在房间内
2. Host 在局间移除该玩家
3. 房间继续正常开始下一手

## 6. 功能范围

### 6.1 已实现功能（In Scope - 当前版本）

#### A. 房间与身份
- 通过 `room` 名称定位房间
- 房间内第一个进入玩家为 Host
- 展示当前 Host 信息
- Host 离开后自动转移 Host
- 支持手动退出房间
- 空房间自动销毁
- Host 可主动解散房间

#### B. 玩家状态
- 登录信息录入：房间名、UID、昵称、买入码量
- 本地持久化登录身份与回填
- 支持切换身份与清除本地身份
- 支持观众席机制
- 支持下局入座
- 支持换座

#### C. 牌局逻辑
- 经典德州扑克规则
- 短牌模式规则
- 发牌、翻牌、转牌、河牌、摊牌
- 弃牌、过牌、跟注、下注、加注、all-in
- fold 胜利结算
- showdown 自动亮牌
- win by fold 后可手动亮牌
- side pot 与多家 all-in 结算
- 两人局与多人局下注顺位处理

#### D. UI 与交互
- 黑白涂鸦风牌桌风格
- PC / Mobile 响应式适配
- 当前行动横幅提示
- 当前行动位强高亮
- 移动端底部固定操作区
- 结算信息展示：收池金额 / 净赢金额

#### E. 运维与部署
- 后端 Docker 构建
- 前端 Docker 构建
- Azure Container Apps 后端部署能力
- 前端静态托管部署能力

### 6.2 当前不在范围内（Out of Scope）

- 用户账号系统 / 密码登录 / OAuth
- 持久化数据库与战绩系统
- 支付、充值、提现、真钱玩法
- 锦标赛、多桌大厅、匹配系统
- 语音聊天、好友系统、邀请系统
- 断线自动托管策略
- 管理后台
- 反作弊、风控、日志审计系统

## 7. 核心规则说明

### 7.1 Classic 模式
- 多人局翻前从 `BB` 左手第一位开始
- 翻后从 `BTN` 左手第一活跃位开始
- 两人局例外：
  - 按钮位 = 小盲
  - 翻前按钮位先动
  - 翻后大盲先动

### 7.2 Short Deck 模式
- 移除 `2 / 3 / 4 / 5`
- 同花大于葫芦
- 每一轮下注均从 `SB` 开始

## 8. 页面结构

### 8.1 登录页
- 房间名输入
- UID 输入
- 昵称输入
- 买入码量输入
- 进入房间按钮
- 清除本地身份按钮

### 8.2 牌桌页
- 顶部：房间、昵称、UID、Host、连接状态、模式
- 中部：底池、轮次提示、公共牌、玩家座位
- 侧边 / 底部：Host 控制、动作按钮、Raise/Bet 输入
- 结算区：赢家、收池金额、净赢金额
- 观众区：观众列表与下局加入提示

## 9. 关键交互要求

### 9.1 当前行动提示
- 当前行动人必须能在 1 秒内被识别
- 自己回合必须有明显提示，不依赖用户细看座位变化

### 9.2 移动端操作体验
- 玩家在手机端应尽可能在同一视口内看到：
  - 当前阶段
  - 自己手牌
  - 操作按钮
- 尽量减少上下滚动

### 9.3 房主控制
- Host 才能修改模式、开始牌局、重开牌局、解散房间、移除玩家
- 局间移除玩家，避免破坏进行中的手牌

## 10. 技术架构概述

### 10.1 前端
- React
- TypeScript
- Vite
- 单页面应用（SPA）

### 10.2 后端
- Go
- `engine`：牌局规则与结算
- `room`：房间、Host、Session、玩家连接
- `network`：SSE 推送与 Action 路由

### 10.3 通信方式
- 下行：`GET /events`（SSE）
- 上行：`POST /action`

## 11. 约束与风险

### 11.1 当前技术约束
- 房间状态当前存储于内存中
- 后端应以单实例运行，避免多实例状态不一致
- 容器重启后房间状态会丢失

### 11.2 风险
- 若后续要支持水平扩展，需要引入 Redis / DB 做状态共享
- 若后续要支持正式产品化，需要补充账号、数据持久化与风控系统

## 12. 测试重点

- 2 人 / 3 人 / 4 人 / 多人下注顺位
- Classic 与 Short Deck 两种模式切换
- 多家 all-in 与边池结算

- Host 权限是否正确限制
- 移动端操作区是否被遮挡
- 登录回填与刷新重连是否稳定

## 14. 验收标准（当前版本）

满足以下条件即认为当前阶段程序可验收：

- 多名玩家可以通过同一房间名加入同一牌桌
- Host 可以控制开局、重开、解散、局间移除玩家
- Classic 模式下注顺位正确
- Short Deck 模式规则与顺位正确
- 多家 all-in 与边池结算正确
- 手机端可以完成一整局对局且操作不需要频繁滚动
- 登录信息可在刷新后回填
- 前后端可通过 Docker 构建并部署
